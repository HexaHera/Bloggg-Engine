Usually, bootstrap processes involve the concept of <em>stages</em>, where a compiler and associated artifacts from a stage is used to build the compiler of the next stage. For example, a stage 2 compiler is built from a stage 1 compiler and associated stage 1 artifacts. The stage number is named after the compiler in question.</p> <h2 id="bootstrapping-rust"><a class="anchor" href="#bootstrapping-rust" aria-hidden="true"></a> Bootstrapping <code>rustc</code></h2> <p>Building a C++ compiler toolchain like <a href="https://gcc.gnu.org/install/build.html"><code>gcc</code></a> or <a href="https://llvm.org/docs/AdvancedBuilds.html#bootstrap-builds"><code>clang</code></a> doesn't usually involving building their standard libraries from source, but instead will link against a pre-built standard library across all bootstrap stages. Typically, this specified standard library is the system standard library.<sup class="footnote-reference" id="fr-adapted-1"><a href="#fn-adapted">1</a></sup></p> <p>The Rust toolchain as a whole, both currently and after this change, supports being built with just two minor versions of Rust: the previous version and its own version. For example, Rust 1.85.1 requires one of 1.84.0, 1.85.0, or 1.85.1 toolchains to build its source code into the 1.85.1 distributed artifacts.</p> <p>Rust has historically always built the majority of in-tree source code against the in-tree standard library's API. This meant that the in-tree copy of the standard library had to support being built with both the previous release (e.g., 1.84) and the new release (e.g., 1.85), requiring extensive <code>cfg(bootstrap)</code> annotations to be compatible with the unstable API of the compiler (primarily changes in intrinsics and lang items).</p> <p>The choice to require std to be built with two different compiler toolchains is largely historical. We don't have specific rationale available, but expect it to have made more sense when the standard library API was rapidly evolving and the compiler was unable to use crates.io dependencies for functionality.</p> <h2 id="terminology"><a class="anchor" href="#terminology" aria-hidden="true"></a> Terminology</h2> <p><img src="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/./naive-bootstrap-stage.svg" alt="Naive bootstrap stage" /></p> <p>Let us call these <em>intra</em>-stage steps of stage <code>N - 1</code> compiler building stage <code>N - 1</code> standard library a <strong>bootstrap sequence</strong>, i.e. the sequence of intra-stage steps involved in producing stage <code>N - 1</code> artifacts required to build the stage <code>N</code> compiler.</p> <p><img src="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/./naive-bootstrap-stage-bootstrap-sequence.svg" alt="Bootstrap sequence" /></p> <p>We must also have a base case, a starting compiler, to build newer compilers with. Indeed, the <em>initial</em> compiler is also called the <em>stage 0 compiler</em>. The <em>initial</em> bootstrap stage is called <em>stage 0</em>.</p> <h1 id="motivation"><a class="anchor" href="#motivation" aria-hidden="true"></a> Motivation</h1> <p>But the naive model we presented above isn't complete.</p> <p><em>Rust</em> has elected a design choice where the compiler, <code>rustc</code>, and the standard library ("std") are tightly coupled. <em>Intrinsics</em> and <em>lang items</em> form a broad interface between the compiler and the standard library. When intrinsics or lang items are modified, both sides need to be adjusted.</p> <p>Currently, the standard library must support being built with two different compilers, the in-tree compiler and the initial stage 0 compiler<sup class="footnote-reference" id="fr-initial-compiler-1"><a href="#fn-initial-compiler">2</a></sup>. All such changes to intrinsics and lang items thus need to use <code>cfg(bootstrap)</code> to gate code that can be built by the in-tree compiler vs the stage 0 compiler. This causes a lot of churn for contributors wanting to introduce, modify or remove intrinsics and lang items (particularly when creating new releases).</p> <p>The <a href="https://github.com/rust-lang/rust/pull/119899">stage 0 bootstrap sequence redesign</a> aims to mitigate such churn and implementation complexity in the standard library by having the standard library only support <em>one</em> version of the compiler.</p> <p>To better understand this redesign, we will:</p> <ol> <li><a href="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/#current-model">Explain how the current stage 0 bootstrapping sequence works</a>, and</li> <li><a href="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/#new-model">Explain how the new stage 0 bootstrapping sequence works after the redesign</a>, and</li> <li><a href="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/#better">Explain the benefits of the redesigned stage 0 bootstrapping sequence</a>.</li> </ol> <h1 id="current-model"><a class="anchor" href="#current-model" aria-hidden="true"></a> The current stage 0 bootstrap sequence</h1> <p><img src="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/./stage0-current.svg" alt="Current stage 0 bootstrap sequence" /></p> <p>Currently, <a href="https://rustc-dev-guide.rust-lang.org/building/bootstrapping/intro.html">bootstrap</a> downloads a pre-built beta rustc as the initial compiler (stage 0 rustc).</p> <ul> <li>To produce a stage 1 rustc, we need to produce a stage 0 std. This stage 0 std is built from in-tree standard library sources with <code>cfg(bootstrap)</code> active.</li> <li>To produce a stage 2 rustc, we need to produce a stage 1 std. However, this is where things get weird. The stage 1 std is built from the <em>same</em> in-tree standard library sources (but with <code>cfg(bootstrap)</code> inactive). This is what we meant by "the standard library has to support being buildable by two compiler versions."</li> </ul> <h1 id="new-model"><a class="anchor" href="#new-model" aria-hidden="true"></a> The redesigned stage 0 bootstrap sequence</h1> <p><img src="https://blog.rust-lang.org/inside-rust/2025/05/29/redesigning-the-initial-bootstrap-sequence/./stage0-next.svg" alt="Redesigned stage 0 bootstrap sequence" /></p> <p>In the <a href="https://github.com/rust-lang/rust/pull/119899">redesigned stage 0 bootstrapping sequence</a> we instead download <em>both</em> the pre-built beta rustc as the stage 0 compiler, and the pre-built beta std as the stage 0 std, <em>instead</em> of building stage 0 std from in-tree sources.</p> <ul> <li>When producing a stage 1 rustc, we already have the stage 0 std (as it is the pre-built std).</li> <li>When producing a stage 2 rustc, the stage 1 std is then the std built from in-tree std sources with the stage 1 rustc.</li> </ul> <h1 id="better"><a class="anchor" href="#better" aria-hidden="true"></a> Why is the redesigned stage 0 bootstrap sequence better?</h1> <p>There are several benefits of the redesigned stage 0 bootstrap sequence:</p> <ol> <li>We no longer have to use <code>cfg(bootstrap)</code> in the standard library sources for intrinsics and lang items to distinguish when being built by the beta rustc vs the in-tree rustc, because the standard library now only has to be buildable by exactly one compiler version (the current stage rustc).</li> <li>Rebasing no longer has to rebuild everything. As the standard library is now built <em>after</em> the compiler instead of before, all dependencies and compiler crates that are unmodified can be cached when switching <code>git</code> branches. It's only necessary to rebuild everything after bootstrap bumps every 6 weeks as part of the release cycle.</li> <li>Modifying the standard library no longer has to rebuild everything. After the redesign, it is now possible to add documentation comments to functions in the standard library without having to rebuild the compiler and then also rebuild std a second time. <code>--keep-stage-std=0</code> is no longer needed.</li> <li>It aligns better with how other rust programs are built. We no longer have a "strange" setup where the stage 1 compiler was built from a <em>beta</em> rustc with an <em>in-tree</em> std. Now, the stage 1 compiler is built from a beta rustc and a beta std.</li> </ol> <h1 id="in-terms-of-bootstrap-invocations-and-bootstrap-config-what-does-this-redesign-mean"><a class="anchor" href="#in-terms-of-bootstrap-invocations-and-bootstrap-config-what-does-this-redesign-mean" aria-hidden="true"></a> In terms of bootstrap invocations and bootstrap config, what does this redesign mean?</h1> <p>The minimum stage to check, build and test the standard library is now stage 1. <code>./x {check,build,test} library --stage=0</code> are now no-ops; switch to <code>--stage 1</code> instead. <code>--keep-stage-std=0</code> is a no-op.</p> <p>For <code>profile = "library"</code> users, like aforementioned, the default check, build, and test stage are now bumped to 1. <code>download-rustc = "if-unchanged"</code> is enabled by default, which downloads a pre-built CI rustc instead of building the compiler if there are no compiler changes, allowing you to build the standard library without building the compiler.</p> <h1 id="what-does-this-mean-for-contributors-working-on-the-standard-library-and-the-compiler"><a class="anchor" href="#what-does-this-mean-for-contributors-working-on-the-standard-library-and-the-compiler" aria-hidden="true"></a> What does this mean for contributors working on the standard library and the compiler?</h1> <ul> <li>Contributors will now no longer need to use <code>cfg(bootstrap)</code> for intrinsics and lang items.</li> <li>Contributors may (rarely) need to use <code>cfg(bootstrap)</code> in compiler code if they wish to experiment with unstable library features <sup class="footnote-reference" id="fr-dogfood-unstable-lib-features-1"><a href="#fn-dogfood-unstable-lib-features">3</a></sup></li> </ul> <h1 id="faqs"><a class="anchor" href="#faqs" aria-hidden="true"></a> Frequently asked questions (FAQs)</h1> <h2 id="faqs-shift-cfg-bootstrap"><a class="anchor" href="#faqs-shift-cfg-bootstrap" aria-hidden="true"></a> Doesn't this just shift <code>cfg(bootstrap)</code> from library code to compiler code?</h2> <p>Not quite. <code>cfg(bootstrap)</code> usage in standard library code for using new intrinsics / lang items (as in the current bootstrap sequence) is much more common than potential <code>cfg(bootstrap)</code> usage in compiler code for experimenting with unstable library features (as in the redesigned bootstrap sequence). This is because the standard library must be changed for new compiler-provided lang items and intrinsics, but the compiler does not (need to) depend on recently added standard library APIs.</p> <p>Additionally, the compiler only needs to add <code>cfg(bootstrap)</code> for anything in the standard library that has changed its unstable API and which is used in the compiler.</p> <blockquote> <p><em>Example: Implementing a trait solving feature which requires adding core lang items</em></p> <p>This will involve adding a new lang item in the compiler (e.g. <a href="https://github.com/rust-lang/rust/blob/5af801b687e6e8b860ae970e725c8b9a3820d0ce/compiler/rustc_hir/src/lang_items.rs#L165"><code>compiler/rustc_hir/src/lang_items.rs</code></a>) and the standard library. Prior to the redesign, the usage of the lang item in the standard library requires <code>cfg(not(bootstrap))</code> since the beta compiler does not know about the new lang item. Recall that the standard library has to support being built by both the beta compiler and the in-tree compiler! After the redesign, <code>cfg(not(bootstrap))</code> usage of the lang item in the standard library is not needed since the standard library is only buildable by the in-tree compiler that adds the new lang item.</p> <p>A stage 2 compiler is <strong>not</strong> required to test the new feature, as the stage 1 library using the lang item is built by the stage 1 compiler, which is the compiler where the new lang item is added!</p> </blockquote> <h2 id="will-this-affect-distribution-packagers"><a class="anchor" href="#will-this-affect-distribution-packagers" aria-hidden="true"></a> Will this affect distribution packagers?</h2> <p>No, this change will not have observable changes for distribution packagers. Distribution packagers will still be able to build e.g. 1.85.0 distribution artifacts with a 1.84.0, 1.84.1 or 1.85.0 toolchain.</p> <h1 id="questions-feedback-bugs"><a class="anchor" href="#questions-feedback-bugs" aria-hidden="true"></a> Questions, feedback, bugs?</h1> <p>You can leave a comment in the <a href="https://rust-lang.zulipchat.com/#narrow/channel/326414-t-infra.2Fbootstrap/topic/Stage.200.20std.20redesign.20support.20thread/with/515096924">zulip support thread for the initial bootstrap sequence redesign effort</a></p> <footer class="footnotes"> <ol class="footnotes-list"> <li id="fn-adapted"> <p>Much of this is adapted from jyn's excellent blog post <a href="https://jyn.dev/bootstrapping-rust-in-2023/"><em>Why is Rust's build system uniquely hard to use?</em></a>. <a href="#fr-adapted-1">↩</a></p> </li> <li id="fn-initial-compiler"> <p>For the vast majority of contributors, the stage 0 "initial" compiler is going to be the beta compiler. The example config file <code>config.example.toml</code> has also been renamed to <code>bootstrap.example.toml</code>.</p> <p><code>config.toml</code> is temporarily accepted as fallback config file to mitigate migration churn across different checkouts, but will be phased out in the future.</p> <h3 id="compiletest-now-enforces-stricter-parsing-of-diagnostics-kinds"><a class="anchor" href="#compiletest-now-enforces-stricter-parsing-of-diagnostics-kinds" aria-hidden="true"></a> <code>compiletest</code> now enforces stricter parsing of diagnostics kinds</h3> <p>Previously, <code>compiletest</code> was very lax in the casing of diagnostic kinds (i.e. the <code>ERROR</code>/<code>HELP</code> portion of error annotations). This mean that annotations such as</p> <pre data-lang="rs" class="language-rs z-code"><code class="language-rs" data-lang="rs"><span class="z-source z-rust"><span class="z-comment z-line z-double-slash z-rust"><span class="z-punctuation z-definition z-comment z-rust">//</span>~ Error xxx </span></span><span class="z-source z-rust"><span class="z-comment z-line z-double-slash z-rust"><span class="z-punctuation z-definition z-comment z-rust">//</span>~ Error: xxx </span></span></code></pre> <p>were also accepted compared to the more common <code>//~ ERROR</code> or <code>//~ error</code> forms.</p> <p>Eventually, we would like to canonicalize the error annotations into one form. To make this transition less abrupt, as an intermediate step, compiletest will now:</p> <ul> <li>Enforce that error annotation diagnostic kinds must either be full caps (e.g. <code>ERROR</code>) or lowercase (e.g. <code>error</code>). Mixed cases like <code>//~ Error</code> will now be rejected.</li> <li>Empty diagnostics (such as empty <code>//~ NOTE</code>) will no longer be silently ignored. These indicate silly empty diagnostic notes emitted by rustc which should be fixed.</li> </ul> <h3 id="compiletest-now-enforces-that-error-annotations-are-required-even-if-error-pattern-directives-are-used"><a class="anchor" href="#compiletest-now-enforces-that-error-annotations-are-required-even-if-error-pattern-directives-are-used" aria-hidden="true"></a> <code>compiletest</code> now enforces that error annotations are required even if <code>//@ error-pattern</code> directives are used</h3> <p>Related to previous <code>//~?</code> improvements, compiletest will now also <a href="https://github.com/rust-lang/rust/pull/139137">check for error annotations (<code>//~</code>) in tests that use <code>//@ error-pattern</code> to minimize risk of accidentally blessing away diagnostics, including diagnostics without associated line information for the main test file.</a>.</p> <p>In <em>exceptional</em> cases, the test writer may opt out of the error annotation checks via <code>//@ compile-flags: --error-format=human</code>.</p> <h3 id="compiletest-now-allows-opting-in-to-non-exhaustive-matching-of-a-specific-diagnostic-kind-via-new-directive-dont-require-annotations-diagnostic-kind"><a class="anchor" href="#compiletest-now-allows-opting-in-to-non-exhaustive-matching-of-a-specific-diagnostic-kind-via-new-directive-dont-require-annotations-diagnostic-kind" aria-hidden="true"></a> <code>compiletest</code> now allows opting in to non-exhaustive matching of a specific diagnostic kind via new directive <code>//@ dont-require-annotations: DIAGNOSTIC_KIND</code></h3> <p><code>compiletest</code> now has a <a href="https://github.com/rust-lang/rust/pull/139489">new directive <code>//@ dont-require-annotations: DIAGNOSTIC_KIND</code></a> (where <code>DIAGNOSTIC_KIND</code> is e.g. <code>ERROR</code>/<code>HELP</code>/<code>SUGGESTION</code>) to allow opting into non-exhaustive matching for the specified <code>DIAGNOSTIC_KIND</code>. This includes the <code>ERROR</code> diagnostic kind, where it was not possible to opt-out of exhaustive matching previously.</p> <p>This directive should be used with caution and sparingly.</p> <h3 id="compiletest-no-longer-accepts-true-false-as-revision-names"><a class="anchor" href="#compiletest-no-longer-accepts-true-false-as-revision-names" aria-hidden="true"></a> <code>compiletest</code> no longer accepts <code>{true, false}</code> as revision names</h3> <p>In test suites that support revisions, previously <code>//@ revisions: true</code> and <code>//@ revisions: false</code> were accepted as revision names. However, this is confusing because they will be automatically registered as <code>--cfg={true,false}</code> to <code>rustc</code>, but the test writer would have to use <code>cfg(r#true)</code> and <code>cfg(r#false)</code> in the test. Hence, <a href="https://github.com/rust-lang/rust/pull/138692">the <code>{true,false}</code> revision names are no longer permitted by compiletest</a>.</p> <h3 id="compiletest-now-supports-a-needs-crate-type-directive"><a class="anchor" href="#compiletest-now-supports-a-needs-crate-type-directive" aria-hidden="true"></a> <code>compiletest</code> now supports a <code>//@ needs-crate-type</code> directive</h3> <p>Test writers can now use a <a href="https://github.com/rust-lang/rust/pull/139469"><code>//@ needs-crate-type</code> directive</a> to guard test execution based on whether the target platform supports all of the required crate types.</p> <p>The directive accepts a comma-delimited list of valid crate types that are accepted by the <code>rustc --crate-type</code> flag. E.g. <code>//@ needs-crate-type: dylib, proc-macro</code>.</p> <p>Example:</p> <pre data-lang="rs" class="language-rs z-code"><code class="language-rs" data-lang="rs"><span class="z-source passed to <code>//@ {unset,}-{exec,rustc}-env</code> directives</h3> <p><code>compiletest</code> has a <a href="https://github.com/rust-lang/rust/issues/132990">long-standing quirk</a> where</p> <pre This means that now, <code>compiletest</code> will reject rustc <code>--edition</code> flags passed via <code>//@ compile-flags</code> in favor of the <code>//@ edition</code> directive, to ensure that <code>compiletest</code> can override the edition of the test.</p> <p>If the test author needs to exercise the behavior of the <code>--edition</code> rustc flag <em>itself</em>, a <code>run-make</code> test should be used.</p> <h2 id="upcoming-changes"><a class="anchor" href="#upcoming-changes" aria-hidden="true"></a> Upcoming changes</h2> <p>The following changes have not merged yet, but will merge in the near future or we are actively working towards them.</p> <h3 id="compiletest-will-soon-also-treat-suggestion-error-annotations-as-viral"><a class="anchor" href="#compiletest-will-soon-also-treat-suggestion-error-annotations-as-viral" aria-hidden="true"></a> <code>compiletest</code> will soon also treat <code>SUGGESTION</code> error annotations as viral</h3> <p>Previously, if the test writer specifies <code>//~ HELP</code> or <code>//~ NOTE</code>, then <code>compiletest</code> treated them as <em>viral</em> -- if one is specified, then all <code>HELP</code>/<code>NOTE</code> diagnostics of the same diagnostic kind must be exhaustively specified.
