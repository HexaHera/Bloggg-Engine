As a Tier 2 Target, builds will continue to be distributed for both the standard library and the compiler.</p> <h2 id="background"><a class="anchor" href="#background" aria-hidden="true"></a> Background</h2> <p>Rust has supported Windows for a long time, with two different flavors of Windows targets: MSVC-based and GNU-based. MSVC-based targets (for example the most popular Windows target <code>x86_64-pc-windows-msvc</code>) use Microsoft’s native linker and libraries, while GNU-based targets (like <code>i686-pc-windows-gnu</code>) are built entirely from free software components like <code>gcc</code>, <code>ld</code>, and mingw-w64.</p> <p>The major reason to use a GNU-based toolchain instead of the native MSVC-based one is cross-compilation and licensing. <code>link.exe</code> only runs on Windows (barring Wine hacks) and requires a license for commercial usage.</p> <p><code>x86_64-pc-windows-gnu</code> and <code>i686-pc-windows-gnu</code> are currently both Tier 1 with host tools. The <a href="https://doc.rust-lang.org/nightly/rustc/target-tier-policy.html">Target Tier Policy</a> contains more details on what this entails, but the most important part is that tests for these targets are being run on every merged PR. This is the highest level of support we have, and is only used for the most high value targets (the most popular Linux, Windows, and Apple targets).</p> <p>The <code>*-windows-gnu</code> targets currently do not have any dedicated target maintainers. We do not have a lot of expertise for this toolchain, and issues often aren't fixed and cause problems in CI that we have a hard time to debug.</p> <p>The 32-bit version of this target is especially problematic and has significantly less usage than <code>x86_64-pc-windows-gnu</code>, which is why <code>i686-pc-windows-gnu</code> is being demoted to Tier 2.</p> <h2 id="what-changes"><a class="anchor" href="#what-changes" aria-hidden="true"></a> What changes?</h2> <p>After Rust 1.88.0, <code>i686-pc-windows-gnu</code> will now be Tier 2 with host tools. For users, nothing will change immediately. Builds of both the standard library and the compiler will still be distributed by the Rust Project for use via <code>rustup</code> or alternative installation methods.</p> <p>This does mean that this target will likely accumulate bugs faster in the future because of the reduced testing.</p> <h2 id="future"><a class="anchor" href="#future" aria-hidden="true"></a> Future</h2> <p>If no maintainers are found and the <code>*-windows-gnu</code> targets continue causing problems, they may be demoted further. No concrete plans about this have been made yet.</p> <p>If you rely on the <code>*-windows-gnu</code> targets and have expertise in this area, we would be very happy to have you as a target maintainer. You can check the <a href="https://doc.rust-lang.org/nightly/rustc/target-tier-policy.html">Target Tier Policy</a> for what exactly that would entail.</p> <p>For more details on the motivation of the demotion, see <a href="https://rust-lang.github.io/rfcs/3771-demote-i686-pc-windows-gnu.html">RFC 3771</a> which proposed this change.</p> Noratrieb April Project Goals Update 2025-05-26T00:00:00+00:00 2025-05-26T00:00:00+00:00 https://blog.rust-lang.org/2025/05/26/april-project-goals-update/ <p>The Rust project is currently working towards a <a href="https://rust-lang.github.io/rust-project-goals/2025h1/goals.html">slate of 40 project goals</a>, with 3 of them designated as <a href="https://rust-lang.github.io/rust-project-goals/2025h1/goals.html#flagship-goals">Flagship Goals</a>. This will change in a future version of the Rust compiler and the <a href="https://github.com/WebAssembly/tool-conventions/blob/main/BasicCABI.md">official C ABI</a> will be used instead.</p> <p>This post details some history behind this change and the rationale for why it's being announced here, but you can skip straight to <a href="https://blog.rust-lang.org/2025/04/04/c-abi-changes-for-wasm32-unknown-unknown/#am-i-affected">"Am I affected?"</a> as well.</p> <h2 id="history-of-wasm32-unknown-unknown-s-c-abi"><a class="anchor" href="#history-of-wasm32-unknown-unknown-s-c-abi" aria-hidden="true"></a> History of <code>wasm32-unknown-unknown</code>'s C ABI</h2> <p>When the <code>wasm32-unknown-unknown</code> target <a href="https://github.com/rust-lang/rust/pull/45905">was originally added</a> in 2017, not much care was given to the exact definition of the <code>extern "C"</code> ABI at the time. In 2018 <a href="https://github.com/rust-lang/rust/pull/48959">an ABI definition was added just for wasm</a> and the target is still using this definition <a href="https://github.com/rust-lang/rust/blob/78948ac259253ce89effca1e8bb64d16f4684aa4/compiler/rustc_target/src/callconv/wasm.rs#L76-L114">to this day</a>. This definitions has become more and more problematic over time and while some issues have been fixed, the root cause still remains.</p> <p>Notably this ABI definition does not match the <a href="https://github.com/WebAssembly/tool-conventions/blob/main/BasicCABI.md">tool-conventions</a> definition of the C API, which is the current standard for how WebAssembly toolchains should talk to one another. Originally this non-standard definition was used for all WebAssembly based targets except Emscripten, but <a href="https://github.com/rust-lang/rust/pull/79998">this changed in 2021</a> where the WASI targets for Rust use a corrected ABI definition. Still, however, the non-standard definition remained in use for <code>wasm32-unknown-unknown</code>.</p> <p>The time has now come to correct this historical mistake and the Rust compiler will soon be using a correct ABI definition for the <code>wasm32-unknown-unknown</code> target. This means, however, that generated WebAssembly binaries will be different than before.</p> <h2 id="what-is-a-webassembly-c-abi"><a class="anchor" href="#what-is-a-webassembly-c-abi" aria-hidden="true"></a> What is a WebAssembly C ABI?</h2> <p>The definition of an ABI answers questions along the lines of:</p> <ul> <li>What registers are arguments passed in?</li> <li>What registers are results passed in?</li> <li>How is a 128-bit integers passed as an argument?</li> <li>How is a <code>union</code> passed as a return value?</li> <li>When are parameters passed through memory instead of registers?</li> <li>What is the size and alignment of a type in memory?</li> </ul> <p>For WebAssembly these answers are a little different than native platforms. For example, WebAssembly does not have physical registers and functions must all be annotated with a type. What WebAssembly does have is types such as <code>i32</code>, <code>i64</code>, <code>f32</code>, and <code>f64</code>. This means that for WebAssembly an ABI needs to define how to represent values in these types.</p> <p>This is where the <a href="https://github.com/WebAssembly/tool-conventions/blob/main/BasicCABI.md">tool-conventions</a> document comes in. That document provides a definition for how to represent primitives in C in the WebAssembly format, and additionally how function signatures in C are mapped to function signatures in WebAssembly. For example a Rust <code>u32</code> is represented by a WebAssembly <code>i32</code> and is passed directly as a parameter as a function argument. If the Rust structure <code>#[repr(C)] struct Pair(f32, f64)</code> is returned from a function then a return pointer is used which must have alignment 8 and size of 16 bytes.</p> <p>In essence, the WebAssembly C ABI is acting as a bridge between C's type system and the WebAssembly type system. This includes details such as in-memory layouts and translations of a C function signature to a WebAssembly function signature.</p> <h2 id="how-is-wasm32-unknown-unknown-non-standard"><a class="anchor" href="#how-is-wasm32-unknown-unknown-non-standard" aria-hidden="true"></a> How is <code>wasm32-unknown-unknown</code> non-standard?</h2> <p>Despite the ABI definition today being non-standard, many aspects of it are still the same as what <a href="https://github.com/WebAssembly/tool-conventions/blob/main/BasicCABI.md">tool-conventions</a> specifies. Why then has the ABI not been fixed like with WASI? The main reason originally for this was the <a href="https://github.com/rustwasm/wasm-bindgen">wasm-bindgen project</a>.</p> <p>In <code>wasm-bindgen</code> the goal is to make it easy to integrate Rust into a web browser with WebAssembly. JavaScript is used to interact with host APIs and the Rust module itself. Naturally, this communication touches on a lot of ABI details! The problem was that <code>wasm-bindgen</code> relied on the above example, specifically having <code>Pair</code> "splatted" across arguments instead of passed indirectly. The generated JS wouldn't work correctly if the argument was passed in-memory.</p> <p>At the time this was discovered it was found to be significantly difficult to fix <code>wasm-bindgen</code> to not rely on this splatting behavior. At the time it also wasn't thought to be a widespread issue nor was it costly for the compiler to have a non-standard ABI. Over the years though the pressure has mounted. The Rust compiler is carrying an <a href="https://github.com/rust-lang/rust/issues/115666">ever-growing list of hacks</a> to work around the non-standard C ABI on <code>wasm32-unknown-unknown</code>. Additionally more projects have started to rely on this "splatting" behavior and the risk has gotten greater that there are more unknown projects relying on the non-standard behavior.</p> <p>In late 2023 <a href="https://github.com/rustwasm/wasm-bindgen/pull/3595">the wasm-bindgen project fixed bindings generation</a> to be unaffected by the transition to the standard definition of <code>extern "C"</code>. In the following months a <a href="https://github.com/rust-lang/rust/pull/117918">future-incompat lint was added to rustc</a> to specifically migrate users of old <code>wasm-bindgen</code> versions to a "fixed" version. This was in anticipation of changing the ABI of <code>wasm32-unknown-unknown</code> once enough time had passed. Since early 2025 users of old <code>wasm-bindgen</code> versions <a href="https://github.com/rust-lang/rust/pull/133951">will now receive a hard error</a> asking them to upgrade.</p> <p>Despite all this heroic effort done by contributors, however, it has now come to light that there are more projects than <code>wasm-bindgen</code> relying on this non-standard ABI definition. Consequently this blog post is intended to serve as a notice to other users on <code>wasm32-unknown-unknown</code> that the ABI break is upcoming and projects may need to be changed.</p> <h2 id="am-i-affected"><a class="anchor" href="#am-i-affected" aria-hidden="true"></a> Am I affected?</h2> <p>If you don't use the <code>wasm32-unknown-unknown</code> target, you are not affected by this change. If you don't use <code>extern "C"</code> on the <code>wasm32-unknown-unknown</code> target, you are also not affected. If you fall into this bucket, however, you may be affected!</p> <p>To determine the impact to your project there are a few tools at your disposal:</p> <ul> <li>A new <a href="https://github.com/rust-lang/rust/pull/138601">future-incompat warning</a> has been added to the Rust compiler which will issue a warning if it detects a signature that will change when the ABI is changed.</li> <li>In 2023 a <a href="https://github.com/rust-lang/rust/pull/117919"><code>-Zwasm-c-abi=(legacy|spec)</code> flag was added</a> to the Rust compiler. This defaults to <code>-Zwasm-c-abi=legacy</code>, the non-standard definition. Code can use <code>-Zwasm-c-abi=spec</code> to use the standard definition of the C ABI for a crate to test out if changes work.</li> </ul> <p>The best way to test your crate is to compile with <code>nightly-2025-03-27</code> or later, ensure there are no warnings, and then test your project still works with <code>-Zwasm-c-abi=spec</code>. If all that passes then you're good to go and the upcoming change to the C ABI will not affect your project.</p> <h2 id="i-m-affected-now-what"><a class="anchor" href="#i-m-affected-now-what" aria-hidden="true"></a> I'm affected, now what?</h2> <p>So you're using <code>wasm32-unknown-unknown</code>, you're using <code>extern "C"</code>, and the nightly compiler is giving you warnings. Additionally your project is broken when compiled with<code> -Zwasm-c-abi=spec</code>. What now?</p> <p>At this time this will unfortunately be a somewhat rough transition period for you. There are a few options at your disposal but they all have their downsides:</p> <ol> <li> <p>Pin your Rust compiler version to the current stable, don't update until the ABI has changed. This means that you won't get any compiler warnings (as old compilers don't warn) and additionally you won't get broken when the ABI changes (as you're not changing compilers). Eventually when you update to a stable compiler with <code>-Zwasm-c-abi=spec</code> as the default you'll have to port your JS or bindings to work with the new ABI.</p> </li> <li> <p>Update to Rust nightly as your compiler and pass <code>-Zwasm-c-abi=spec</code>. This is front-loading the work required in (1) for your target. You can get your project compatible with <code>-Zwasm-c-abi=spec</code> today. The downside of this approach is that your project will only work with a nightly compiler and <code>-Zwasm-c-abi=spec</code> and you won't be able to use stable until the default is switched.</p> </li> <li> <p>Update your project to not rely on the non-standard behavior of <code>-Zwasm-c-abi=legacy</code>. This involves, for example, not passing structs-by-value in parameters. You can pass <code>&amp;Pair</code> above, for example, instead of <code>Pair</code>. This is similar to (2) above where the work is done immediately to update a project but has the benefit of continuing to work on stable Rust. The downside of this, however, is that you may not be able to easily change or update your C ABI in some situations.</p> </li> <li> <p>Update to Rust nightly as your compiler and pass <code>-Zwasm-c-abi=legacy</code>. This will silence compiler warnings for now but be aware that the ABI will still change in the future and the <code>-Zwasm-c-abi=legacy</code> option will be removed entirely. When the <code>-Zwasm-c-abi=legacy</code> option is removed the only option will be the standard C ABI, what <code>-Zwasm-c-abi=spec</code> today enables.</p> </li> </ol> <p>If you have uncertainties, questions, or difficulties, feel free to reach out on <a href="https://github.com/rust-lang/rust/issues/138762">the tracking issue for the future-incompat warning</a> or on Zulip.</p> <h2 id="timeline-of-abi-changes"><a class="anchor" href="#timeline-of-abi-changes" aria-hidden="true"></a> Timeline of ABI changes</h2> <p>At this time there is not an exact timeline of how the default ABI is going to change. It's expected to take on the order of 3-6 months, however, and will look something roughly like this:</p> <ul> <li>2025 March: (soon) - a <a href="https://github.com/rust-lang/rust/pull/138601">future-incompat warning</a> will be added to the compiler to warn projects if they're affected by this ABI change.</li> <li>2025-05-15: this future-incompat warning will reach the stable Rust channel as 1.87.0.</li> <li>2025 Summer: (ish) - the <code>-Zwasm-c-abi</code> flag will be removed from the compiler and the <code>legacy</code> option will be entirely removed.</li> </ul> <p>Exactly when <code>-Zwasm-c-abi</code> is removed will depend on feedback from the community and whether the future-incompat warning triggers much. It's hoped that soon after the Rust 1.87.0 is stable, though, that the old legacy ABI behavior can be removed.</p> Alex Crichton
